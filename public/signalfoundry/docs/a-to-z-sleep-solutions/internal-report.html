Schema Operator Brief

Internal-only schema diagnosis, implementation rationale, and delivery checklist. Keep this detail out of the customer-facing Verity Scan unless translated into plain-language findings.

Diagnosis
Current JSON-LD types detected: BreadcrumbList, EntryPoint, ImageObject, ListItem, Organization, ReadAction, SearchAction, WebPage, WebSite.The internal question is not only whether schema exists, but whether the business, website, page, person, and service entities are distinct and connected cleanly.Preserve useful existing entity signals, but reduce ambiguity by keeping one stable practice/business entity and linking related people, pages, and services to it.Visible service or treatment content should be represented only when the service is actually supported by page copy or a dedicated treatment page.The business should be typed more specifically as LocalBusiness + Service instead of relying on generic website or organization context alone.
Before / After Rationale
Before: the workbench captured raw current schema and generated a thin starter object that did not fully explain the entity model.After: the proposed schema uses a connected @graph with one canonical business/practice entity, one WebSite entity, one homepage WebPage entity, and service entities where appropriate.The goal is to make the same real-world business easier to reconcile across pages, search systems, and AI answer engines without inventing unverified facts.
Implementation Notes
Find the theme, plugin, or tag-manager source that currently emits JSON-LD before removing or replacing existing schema.Keep the canonical business @id stable and reuse it anywhere pages, services, reviews, videos, people, or offers need to point back to the business.Do not add sameAs, logo, geo coordinates, licenses, credentials, awards, reviews, or doctor/practitioner claims unless each value is verified from an approved source.For service nodes, include only services that are visible on the live site and preferably supported by a dedicated page or section.Deploy the schema as one clean JSON-LD graph or a small coordinated set of graphs; avoid multiple plugins emitting conflicting LocalBusiness, Organization, or practitioner records.
Validation Checklist
Validate the final JSON-LD with Schema.org's validator and Google Rich Results Test where relevant.Confirm name, address, phone, URL, and business type against the live website before delivery.View page source after deployment and confirm the old duplicate or conflicting schema records are removed or reconciled.Check that every @id reference resolves to another node in the graph or a stable canonical URL.Confirm the customer-facing report stays plain-language and does not expose raw JSON-LD, local paths, or operator implementation notes.
